# CLI Interface Specification (09.01_ui_ux_atomic_design.md)

**Purpose:** Map the visual atoms, color standardization, and typographic readability to ensure that the terminal text interface offers an immersive, strict, and consistent user experience (UX).

---

## 1. Responsive Approach and Screen Limits
> How does the interface adapt to the changing width of the Terminal and ensure continuous readability?
- **Delegation to the Host Client (Word-Wrap Agnostic):** The Skill's architecture does not inject artificial line breaks (e.g., hardcoded limits of 80 or 100 columns). Responsiveness is strictly delegated to the rendering engine the user is utilizing (Terminal Emulator, TUI, or IDE's native UI). This approach ensures that horizontally sensitive elements (like structured Markdown Tables and native Code Blocks) do not violently "break" when undergoing screen shrinkage.

## 2. Structure and Hierarchy (CLI Grid)
> How do we structure text blocks, bullet points, and spacing (line breaks)?
To combat the visual fatigue inherent in plain text interfaces, the vertical density of the Output adopts the following structural hierarchy premises:
- **Abrupt Context Separation:** It is structurally mandatory to inject a physical "divider" composed of `Line Break` + `---` (Horizontal Rule) + `Line Break` between the block ending the feedback from the previous answer and the block presenting the new question.
- **Highlighted Feedback:** The *Acknowledgment* paragraph (recognizing the previous answer) must be formatted in a visually distinct way (e.g., use of Blockquotes `> ` or specific colors) so it doesn't mix with the new cognitive payload.
- **Breathable and Scannable Blocks:** The system actively rejects dense blocks of continuous prose. Explanations and flows must be broken up using double spacing (blank lines) and extensive use of Bullet points, leveraging reading speed and the user's visual scanning.

## 3. Colors and Typography (Design Tokens)
> Specification of the semantic color scheme via Markdown markup.
To ensure maximum visual accessibility, the color palette acts strictly **Theme-Agnostic** (agnostic to the user's theme). The engine does *not* spit out hardcoded hexadecimal or ANSI codes that would be illegible on a White background (Light Mode) opposing a Black background (Dark Mode). Instead, we inject native *Semantic Meaning* in Markdown, delegating the contrast intelligence to the terminal emulator's own Color Theme.

The structural matrix of Design Tokens:
- **Primary Highlight Token (`## Headers` or `**bold**`):** Typically rendered in Yellow/Gold. Triggers primary attention for Question Titles and critical architectural concepts.
- **Technical Element Token (Backticks `` ` ``):** Typically rendered in Cyan/Blue. Strictly delimits CLI commands, snippets, routes, and file system paths.
- **Metadata Token (`<brackets>` or `'quotes'`):** Typically rendered in Orange/Purple. Used to scope substitution variables and hard definitions.
- **Confirmation Token (Blockquotes `> `):** Typically rendered in Green/Italic-Gray. Anchored exclusively for positive Feedback blocks and isolated validations.

## 4. Interface States and Feedback (Micro-interactions)
> How does the system signal Loading, Success, or Error without a static graphical interface?
- **Loading and Processing (Pass-Through):** The architecture does not pollute the output with artificial `[Thinking...]` messages or ASCII spinning wheels. The blocking read state is delegated entirely to the LLM's host interface (which usually already handles the data stream or displays Tool Calls panels natively in progress).
- **Continuous Progress Feedback (On-Demand):** To avoid abusive spending of Output Tokens and prevent mathematical hallucinations by the LLM, the tool does **not** inject percentage progress bars (`[███-------] 30%`) with each interaction round. The overview of Topics and Methodologies advancement is isolated and rendered exclusively on demand via the `/interview status` command.
- **Errors and Exceptions:** Blocking messages or scope deviations do not have "error pages". They are prefixed with short and direct semantic flags (e.g., `[OUT_OF_SCOPE]`) and lock the synchronous pipeline requiring user rewriting.

## 5. Cognitive Accessibility (A11y)
> Active practices adopted in the Output to not generate cognitive fatigue or reading overload.
Accessibility in a textual environment transcends the visual and focuses on eradicating the Developer's "cognitive load" and processing anxiety. To achieve this, the architecture rests on three major pillars intrinsic to the Skill:
- **Restricted Cadence (One-by-One):** The absolute Golden Rule that prohibits sending multiple simultaneous questions. Divide and conquer. A single processed problem at a time reduces resolution anxiety.
- **Operational Tone and Conciseness:** The communication imposed on the AI rejects excessive colloquialism ("fat") and empty formalities. The answers are atomic and straight to the point, saving the mental energy of decoding.
- **Constant Scanning Pattern:** The rigorous standardization of the vertical Grid (constant use of bullet points and `---` separators) ensures that the human eye quickly tracks where the answer begins and where the feedback ends, reducing the constant visual search effort.
