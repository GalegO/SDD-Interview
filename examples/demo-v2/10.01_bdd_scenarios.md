# BDD Scenarios (10.01_bdd_scenarios.md)

**Purpose:** Map the strict behavior of the CLI through declarative scenarios in Gherkin syntax, translating business rules into acceptance testing flows.

---

## Scenario 1: Happy Path (Multiple Commands)
> What are the ideal error-free flows for each functional trigger of the architecture?

### Scenario 1.1: Clean Initialization (`/interview new`)
- **Given:** That the User wants to document a project from scratch.
- **When:** They execute the command `/interview new <alias>`.
- **Then:** The Agent must create the directory and the base file `00.00_tracking_changelog.md`.
- **And:** Render the initial message and the 1st methodological question.

### Scenario 1.2: Context-Based Initialization (`/interview docs`)
- **Given:** That the User has old documentation to anchor the project.
- **When:** They execute `/interview docs <docs_path> <alias>`.
- **Then:** The Agent must perform deep ingestion by reading the legacy context.
- **And:** Start the interview cycle formulating questions already calibrated by this context.

### Scenario 1.3: Operational Interaction Cycle (Chat Response)
- **Given:** That the interview is active and a question is pending.
- **When:** The User submits their answer validly.
- **Then:** The Agent must structure and write the technical information in the respective `.md` file.
- **And:** Advance the state pointer in `00.00_changelog.md`.
- **And:** Present the next question maintaining isolated cadence.

### Scenario 1.4: Safe Session Resumption (`/interview resume`)
- **Given:** That the IDE was closed and the User wants to continue an interview.
- **When:** They execute `/interview resume <path>`.
- **Then:** The Agent must reread the `00.00` YAML tree reloading the state machine.
- **And:** Identify the paused topic and re-render the last pending question.

### Scenario 1.5: Passive Audit (`/interview status`)
- **Given:** That the User needs to monitor document coverage.
- **When:** They trigger the analytical dashboard command `/interview status <path>`.
- **Then:** The Agent must scan the internal YAML tree and display the numeric ASCII progress dashboard on the terminal screen.
- **And:** Refrain from generating new files or questions on the main pipeline.

### Scenario 1.6: Active Compliance Audit (`/interview validate`)
- **Given:** That the Architecture team needs to attest to the technical validity of the produced artifacts.
- **When:** The User actively executes `/interview validate <path>`.
- **Then:** The "Auditor" AI must do a complete crawling in the informed folder.
- **And:** Produce and materialize a strict report pointing out modeling flaws or loose ends in the requirements.

### Scenario 1.7: Dynamic Help (`/interview help`)
- **Given:** That the User needs to consult the engine's options and skills.
- **When:** They execute `/interview help`.
- **Then:** The Agent must display the formatted and documented manual of commands.
- **And:** Keep the internal state intact, without forcing flow advancement or disk writing.

## Scenario 2: Logical Exception Handling
> What happens if there is a validation error in the flow or provided data?

### Scenario 2.1: Shallow Answers or "Skips" (Quality Protection)
- **Given:** That the User sends an empty answer (e.g., "I don't know") or aggressively asks to skip the active methodological question.
- **When:** The AI receives the explicit intention to abstain from answering.
- **Then:** The Agent must allow the skip so as not to hurt the fluid experience.
- **And:** It must obligatorily issue a friendly Quality Degradation alert, warning the user that ignoring central steps will directly impair the technical richness of the final documentation.

### Scenario 2.2: Non-existent Resumption Context (Abortive `/interview resume`)
- **Given:** That the User got the folder tree wrong in the VSCode terminal.
- **When:** They execute `/interview resume <path>` in a directory with no history.
- **Then:** The Skill engine must interrupt the LLM trigger.
- **And:** Throw a friendly textual error indicating the absence of the base `00.00` file, avoiding creating hallucinated parallel states.

### Scenario 2.3: Omission of Contractual Parameters
- **Given:** That the User attempts to invoke direct subcommands.
- **When:** They execute triggers like `/interview new`, `/interview docs`, `/interview validate`, `/interview resume`, or `/interview status` forgetting to provide the mandatory variables of alias or folder path.
- **Then:** The graphical interface/CLI must block execution at the edge of the protocol.
- **And:** Return in the console the strict syntax required (e.g., `Expected usage: /interview new <alias>`), interrupting empty processing.
> Critical cases that must be validated via TDD.

## Scenario 3: Concurrency Restrictions and Integrity
> What occurs in scenarios of simultaneous use or physical file concurrency?

### Scenario 3.1: I/O Concurrency (Multiple Instances)
- **Given:** That the inattentive User opens multiple instances of the IDE or terminal tabs in the same project directory.
- **When:** They try to invoke Skill commands and write files simultaneously in these instances.
- **Then:** The tool's architecture will **not** apply systemic protection locks (like LockFiles or operation queues), because it is a simple local Skill and not a complex asynchronous system.
- **And:** The system will allow simultaneous operations assuming a Pass-Through stance, where the operational responsibility and the risk of corrupting the history (or suffering overwrite) is delegated entirely to the User's care.
