# Strategic Design and Bounded Contexts (05.02_ddd_strategic_design.md)

**Purpose:** Model the strategic design of the Skill's architecture, delimiting contexts and standardizing the official vocabulary.

---

## 1. Domain Architectural Decisions
> Describe the strategic decisions to solve the presented problem.
- **100% Flat-File Storage (Markdown):** Strict adoption of the user's file-system as the "state database", eliminating the need for traditional databases and keeping the solution hyper-portable, lightweight, and human-readable.
- **Hosted Execution (Agent-Based):** Absence of its own interface (GUI/TUI) or standalone executable. All logical processing, domain validations, and user interaction are outsourced to the native LLM agent in use.
- **State Control by Sentinel (Changelog):** Centralization of the session's memory in a single static file (`tracking_changelog.md`), ensuring idempotency, continuous transactional tracking, and native support for asynchronous interview resumption (`resume` command).

## 2. Bounded Contexts Involved
> Which Bounded Contexts of the system will be affected or created?
- **Orchestration Context (Engine):** Encompasses the Skill's technical machinery (architecturally represented by the `SKILL.md` file and the `reference/` folder). It is exclusively responsible for the session lifecycle, command interpretation (`new`, `resume`, `validate`), file management (I/O), and state protection (`tracking_changelog.md`). This context is agnostic (blind) to business methodologies.
- **Methodological Context (Interview Framework):** Encompasses the structural business content (represented by the `phases/` and `Templates/` folders). This is the context that understands and executes the methodologies (BMC, MoSCoW, DDD, etc.), dictating the sequence of questions, content validations for each phase, and shaping the final writing in the `.md` artifacts.

## 3. Ubiquitous Language (Local Glossary)
> Specific business terms that everyone should use consistently in this scope.
- **Alias:** The name that christens the project directory and defines the scope of the current interview.
- **Tracking Changelog:** The sentinel file (`00.00_tracking_changelog.md`) that acts as "living memory", tracking the session progress and covered topics.
- **Engine:** The root execution environment (architecturally the `SKILL.md` + `reference/`) that governs the machinery, commands (I/O), and the Skill's lifecycle.
- **Interview Framework (Phases):** The methodological scripts (`phases/`) that dictate business rules and the logical question roadmap.
- **Base Template:** Standardized structural blank files (`Templates/`), consumed as molds to start new artifacts.
- **Final Artifact:** The validated `.md` document, iteratively filled by the AI and consolidated in the user's directory.

## 4. Boundaries and Interactions
> How does this context interact with others? Are there communication APIs between them? Global events?
Communication between the Orchestration Context (Engine) and the Methodological Context (Framework) occurs strictly via **Disk I/O (File System)**, operated by the LLM agent. There are no external APIs or network protocols involved:
- The **Engine** triggers the **Framework** via static reading of scripts (`phases/`) and molds (`Templates/`).
- The **Framework** consolidates its processing by writing directly to disk (Final Artifact generation).
- The global transaction between the two contexts is exclusively synchronized by reading/writing to the central state file (`tracking_changelog.md`).
