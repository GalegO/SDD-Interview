# Contract Specifications (06.02_cqrs_specs.md)

**Purpose:** Map the strict constraints, the golden rules of the LLM's behavior, and the definition of the data (I/O) trafficked in the application.

---

## 1. Specific Business Rules
> Strict conditions, constraints, and mandatory behaviors of the LLM Agent.
- **Rule 1 (Strict Cadence):** The interview must obligatorily flow with only **one question at a time**. It is strictly forbidden to send multiple questions from different topics in the same cycle to avoid subject overlap and loss of focus.
- **Rule 2 (Depth Validation):** The Agent must never passively accept and record shallow information; it is mandatory to challenge the user to deepen the answer with technical rigor.
- **Rule 3 (Transactional Consent):** Advancing topics and writing state to disk can only occur after the collaborative validation and consent of the human.
- **Rule 4 (Anchoring in Docs Mode):** In the pre-existing documentation reading mode (`docs`), the Agent is prohibited from inferring/hallucinating data from its own base; it must propose drafts anchored strictly in the documentation provided by the user.
- **Rule 5 (Write Isolation / Read-Only Source):** The AI is strictly prohibited from writing or modifying any file in the documentation folder provided in `docs` mode (`docs_path`). The source documentation must be operated as **Read-Only**. All artifact writing must occur exclusively in the destination folder of the interview.
- **Rule 6 (Security and Sanitization):** If real credentials, tokens, passwords, or sensitive personal data are identified in the user's prompt, the Agent must abort writing to disk, pause the flow immediately, and demand prior sanitization.
- **Rule 7 (Lazy Loading of Templates):** The Agent is structurally prohibited from reading the *Template* files of methodologies that are not the active target at the moment, preventing premature pollution and useless exhaustion of the reading context window (Memory Limit).

## 2. API Contracts (Inputs and Outputs)
> Specification of formats (YAML, Markdown) and expected data contracts in operations.
In the architecture of this local solution, the "HTTP Endpoints" are replaced by the **CLI Command Routes**, and the "JSON Payloads" are replaced by asynchronous interactions in **Markdown**.

### Endpoint (Command): `/interview new <alias>`
- **Description:** Starts a new interview session in a clean folder within the target directory.
- **Input (Payload):** The name (`alias`) of the new folder and the submission via prompt of the initial setup (language selection and methodological IDs).
- **Output (Success):** The I/O engine creates the folder and materializes the sentinel file `00.00_tracking_changelog.md` with the native `in_progress` status.

### Endpoint (Command): `/interview docs <docs_path> <alias>`
- **Description:** Starts the interview in a hybrid document reading mode (`docs`).
- **Input (Payload):** The absolute reading path (`docs_path`) of the legacy documentation. The engine actively reads the folder and indexes the context, without modifying the source (Rule 5).
- **Output (Success):** Creation of the destination folder with the changelog pointing to the existence of the source.

### Endpoint (Command): `/interview resume <path>`
- **Description:** Restores context and resumes the exact state of a paralyzed interview.
- **Input (Payload):** Absolute path containing the interview. The LLM reloads the `changelog` YAML tree.
- **Output (Success):** The Agent triggers the next pending methodological question on the user's screen.

### Endpoint (Command): `/interview status <path>`
- **Description:** Performs the analytical query of the methodological progress (*Pure Query*).
- **Output (Success):** Returns the visual progress report in the chat window, without triggering disk write/I/O commands.

### Endpoint (Command): `/interview validate <path>`
- **Description:** Triggers the local quality assurance (QA) audit pipeline of the generated project.
- **Output (Success):** Materialized generation of a strict reporting artifact `00_validation_report.md` consolidating methodological integrity.

## 3. Expected Error Codes
> Handling of exceptions and warnings issued by the system or guide.
In the Skill's CLI/LLM ecosystem, exception handling occurs through pauses in execution and systemic alerts thrown in the terminal, ensuring the integrity of the state machine with no relation whatsoever to HTTP protocols:

- **`[INPUT_ERROR]` (Validation Failure):** Triggered on invalid insertions by the user. Ex: Initial selection of methodologies empty or outside the `0-13` scope.
- **`[SECURITY_BLOCK]` (Consent Lock):** Lock on the attempt of destructive overwrite (*Restart*) or generation of files without explicit approval.
- **`[MISSING_STATE]` (Context Deviation):** Non-existent destination folder or absence of the vital `00.00_tracking_changelog.md` file when invoking state-dependent commands (`/resume`, `/status`, `/validate`).
- **`[RATE_LIMIT_EXCEEDED]` (Quota Exceeded):** Request limit exceeded at the LLM provider. I/O is suspended, keeping the changelog ready for resumption via `/resume`.
- **`[PROVIDER_OFFLINE]` (Connection Failure):** Network drop or overload on AI servers. Execution is safely aborted, preserving the last generated artifact.
