# MoSCoW Prioritization (03.01_moscow_prioritization.md)

> [!TIP]
> The MoSCoW method helps align expectations on what will be delivered in the first version (MVP) and what will be left for later.

## 1. Must Have
> What is vital, without which the project/feature cannot be considered done?
- The primary initialization command (e.g., `new` command), responsible for creating a structured plan from scratch through a cadenced question-and-answer chat. This is the core of the Skill.

## 2. Should Have
> What is very important, but not a blocker for the first version (MVP)?
- Implementation of secondary support commands, such as `docs` (documentation access), `resume` (to pick up where left off), and `help` (quick guide). They are huge value aggregators, but the MVP can be viable even before they are perfect.

## 3. Could Have
> What would be a very nice bonus ("Nice to Have") if there is time/resources?
- Advanced tracking commands, such as `status` (to view the interview progress) and `validate` (to check the consistency of answers).
- The optional generation of an extra Markdown document with a commercial/marketing presentation appeal, complementing the Executive Summary vision (98.01).

## 4. Won't Have (Out of Scope)
> What will we definitely not do now (or ever) to maintain focus?
- The Skill will **never** have the purpose of generating source code or any other format other than Markdown (`.md`). Its role is to be purely a functional guide.
- The tool will **never** have its own Graphical User Interface (GUI) or Text User Interface (TUI). It will make exclusive and native use of the platform's interface where the user is operating.
