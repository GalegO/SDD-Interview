**Purpose:** Define and break down the macro context of the project into atomic tasks and clear acceptance criteria.

## 1. Macro Context (Epic)
> Briefly describe which Epic or broader business demand this subtask belongs to.
- This delivery serves the macro Epic of **"Interactive Technical Planning Framework"**, acting as the central engine for organizing and extracting structured requirements.

## 2. Business Vision (CEO Opinion)
> Strategic analysis and business impact generated during the demand conception.
- **Estimated Cost:** N/A (Open-source project developed as a hobby).
- **Expected ROI:** N/A (Focused purely on public utility and technical learning).
- **Priority:** Flexible (Personal development).
- **Business Value:** Delivery of a free, valuable tool to the community, radically simplifying the software planning stage.
- **Urgency:** Low.

## 3. Atomic Subtask Description
> What is the isolated task that needs to be executed? (Must be focused and indivisible).
- Develop the architecture and Markdown scripts responsible for the primary initialization command (`new`), which will act as the main engine of the Skill, guiding the user in creating a plan from scratch through an iterative interview.

## 4. Basic Acceptance Criteria
> List of minimum conditions for the task to be considered completed.
- The initialization command (`new`) must guide the user autonomously and iteratively.
- The secondary support commands (`docs`, `resume`, `help`) must be functional.
- The tracking and auditing utilities (`status`, `validate`) must operate correctly.
- The Skill must be able to export the bonus presentation document with commercial/marketing appeal at the end of the flow.

## 5. Restrictions and Assumptions
> Any technical limitations, dependencies, or assumptions.
- The Skill has as a non-negotiable premise the prohibition of generating any executable code, being strictly focused on creating structured documentation in Markdown (`.md`).
- The tool is restricted to using the native interface of the platform where the user is operating; the creation or adoption of its own GUI/TUI is expressly prohibited.
