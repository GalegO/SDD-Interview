# Test Strategy and TDD (11.01_tdd_test_strategy.md)

**Purpose:** Define the parameters for quality assurance, testability of Markdown artifacts, and validation of the Artificial Intelligence's behavior in response to CLI commands.

---

## 1. Integration and Backend Tests (LLM/CLI)
> How do we test the effectiveness of CLI and AI processing?
- **Adopted Approach (Structured Human QA / Pass-Through):** Due to the highly stochastic (variable) nature of Artificial Intelligence responses and the absence of classic executable code in the repository, the use of predictive mechanized TDD (e.g., `pytest`, `promptfoo`, or Regex evaluating scripts) is structurally vetoed.
- Quality assurance of the "BDD Engine" behavior (validation of triggers like `/new`, `/resume`, and exception handling) is validated entirely through rigorous sessions of **Real-World QA**. The human operator acts as an evaluator, executing the tool in the chat interface and attesting *in loco* to the adherence of responses, exactly simulating the daily production scenario.

## 2. Frontend and UI Tests
> How do we test the terminal reading interface?
- **Adopted Approach (QA and Continuous Visual Approval):** Although the architecture adopts a Pass-Through graphical pattern delegating absolute color rendering to the user's IDE/Terminal, the system is not exempt from Frontend responsibility.
- UI tests are actively performed "by eye" by the operator during the QA cycle. The goal is to strictly audit the syntactic behavior generated by the AI (if it obeys the "Design Atoms"): attesting if the vertical Grid breathes, if the systematic use of `---` prevents blocks from merging, and if the color markers (headers, blockquotes, and backticks) are not generating visual fatigue on the screen.

## 3. Database and State Tests
> How do we evaluate application state storage?
- **Adopted Approach (Physical Visual Audit and Semantic Inspection):** The sanity of our "Database" (physical `.md` files and the state tree in `00.00`) is validated primarily through active observation of the File System. The operator monitors in real-time whether files are being born and updated on disk without corruption.
- Beyond pure systemic I/O persistence, the database test obligatorily requires **Cross-Semantic Inspection**. The evaluator (User) must attest whether the AI Agent managed to perfectly extract and relate the persisted context, checking if the suggestions and the progression of the interview make strict logical sense based on the data it consumed from the "database".

## 4. Native Performance Tests
> How do we ensure the Skill won't consume abusive token/time limits?
- **Adopted Approach (Pass-Through / Architectural Exemption):** Load tests (Stress Testing) and latency tests are strictly classified as Not Applicable (N/A). The tool operates as a passive text contract. Responsibility for managing Timeout limits, network latency, character streaming bottlenecks, and Token costs is entirely delegated to the infrastructure of the LLM provider chosen by the developer and their respective IDE.

## 5. Edge Cases and Compliance
> How do we ensure structural protections and adaptations in the face of chaotic human behavior?
- **Adopted Approach (Scope Resilience and Organic Flexibility):** The stress test (Edge Cases) of the Skill focuses on validating the cognitive elasticity of the AI. QA actively simulates the organic flow of thought of a human developer.
- The test requires forcing a "back-and-forth" behavior: the operator must pull dense conceptual discussions, request the revision of ideas from previous phases based on new insights, temporarily engage in parallel research, and interact without rigidity.
- The Success Criterion is that the Skill withstands the chaos of creativity without being rigid or punitive. It must allow enriching debate, update the history as necessary, and, subtly and organizedly, anchor the discussion back to the original point, maintaining BDD/TDD progression without losing the state machine.
