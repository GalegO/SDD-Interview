# Project Briefing (00.01_initial_discovery_project_briefing.md)

**Objective:** Record the responses from Phase 0 (Initial Discovery), serving as the foundational context and guiding compass for all other phases and technical decisions of the project.

---

## 1. Identification
- **Project Name:** Skill V2 Test
- **Macro Objective:** Test the refactoring and V2 of this Skill

## 2. Technical Scope and Platform
- **Target Platform:** CLI, IDE, AI Agents and Models
- **Initial Tech Stack:** Only Markdown (MD) files for input and output
- **Complexity and Challenges:** Overall low complexity. The biggest challenge is getting users to test across various environments (CLIs, IDEs, models) to ensure solid validation and make the solution as agnostic as possible.

## 3. Identity
- **Branding and Tone:** Collaborative, consultative, and natural, adopting the posture of a **Technical Mentor/Advisor**. Structured interview style, providing specific guidance to stimulate reflection based on previous answers, without leading to false paths. The AI should adopt a critical stance and question the user **only** when noticing deviations from the framework, project structure flaws, or escaping the current scope, explaining "why" and proposing the correct path. Balance between professionalism and fluidity.
- **Additional Communication Rules:** Avoid slang, but keep the document's language light and direct. Do not use excessive formality (avoid "fluff generator" style). The goal is to be clear and guarantee impeccable documentation quality, translating mannerisms into a professional and fluid tone. The critical tone should not make the conversation rigid, appearing only to correct technical mistakes, methodology application failures, or deviations from the already established interview baseline. Vital flow rule: NEVER update the files when the user asks a question or is in doubt; the conversation and debate must continue until the user's explicit OK/acceptance.
